# 勤怠管理アプリ

## 1. 目的

### 誰が困っているか
- **小規模チーム（5〜30名）のリーダー・管理者**が、メンバーの勤怠をExcelや紙で管理しており、集計・確認に毎月多くの時間を費やしている
- **現場スタッフ（メンバー）** が、打刻忘れやシフト申請のたびに口頭・チャットで連絡しており手間がかかっている

### 何に困っているか
- 出退勤の記録が散在し、正確な労働時間が把握できない
- シフト変更や休暇申請の承認フローが属人的で、申請状況が見えない
- 日報の提出・確認が煩雑で、業務内容の振り返りができていない

### どう解決するか
- **ワンタップ打刻**で出退勤を即時記録し、集計を自動化する
- **申請→承認の統一フロー**で、シフト・休暇のやりとりを透明化する
- **日報・カレンダー・ダッシュボード**で、チーム全体の稼働状況を一目で把握できるようにする

## 2. 画面構成（5画面）

### 2-1. 打刻画面
- 出勤・退勤ボタン（ワンタップ操作）
- 現在のステータス表示（未出勤 / 勤務中 / 退勤済）
- 本日の勤務時間リアルタイム表示
- 打刻履歴（直近7日間）

### 2-2. シフト申請画面
- シフト希望日・時間帯の入力フォーム
- 休暇申請（有給・半休・欠勤）
- 申請一覧と現在のステータス（申請中 / 承認 / 却下）
- 却下時の理由表示

### 2-3. カレンダー画面
- 月間カレンダーで勤務実績を色分け表示
  - 出勤日: 青 / 休暇: 緑 / 欠勤: 赤 / 未確定: グレー
- 日付タップで詳細表示（勤務時間・日報リンク）
- admin: チーム全員のカレンダー切替表示

### 2-4. 日報画面
- 当日の業務内容入力（テキスト）
- カテゴリ選択（通常業務 / MTG / 研修 / その他）
- 過去の日報一覧・検索
- admin: メンバーの日報閲覧

### 2-5. ダッシュボード画面
- member: 今月の勤務サマリー（総労働時間・出勤日数・残業時間）
- admin: チーム全体の出勤状況（本日の出勤者数・未出勤者リスト）
- admin: 未処理の承認リクエスト一覧
- admin: 今月のチーム稼働サマリー（平均労働時間・有給消化率）

### レイアウト共通仕様

サイドバー + メインエリアの2カラム構成。サイドバーのメニュークリックで右側のメインエリアが切り替わる。

#### サイドバー
- 幅: 260px（固定）
- 背景色: 青系（Tailwind `bg-blue-900` / テキスト `text-white`）
- 上部にアプリロゴ・アプリ名を配置
- 5つのメニュー項目を縦に並べる
- アクティブなメニューは背景をハイライト（`bg-blue-700`）
- アイコンは `lucide-react`（shadcn/ui 標準）を使用:

| メニュー | アイコン |
|---------|---------|
| 打刻 | `Clock` |
| シフト申請 | `CalendarClock` |
| カレンダー | `CalendarDays` |
| 日報 | `FileText` |
| ダッシュボード | `LayoutDashboard` |

#### メインエリア
- サイドバーの右側に配置（`ml-[260px]` または flex レイアウト）
- 余白はレスポンシブ対応: `p-4 sm:p-6 lg:p-8`
- 最大幅の制限なし（コンテンツに応じて伸縮）

#### モバイル対応（将来拡張）
- 768px 以下でサイドバーを非表示にし、ハンバーガーメニューで開閉
- 初期実装ではデスクトップファーストで進める

## 3. 承認フロー

```
member が申請を作成
       │
       ▼
   ┌────────┐
   │ 申請中  │  ← admin の承認待ち
   └────┬───┘
        │
   admin が確認
        │
   ┌────┴────┐
   ▼         ▼
┌─────┐  ┌─────┐
│ 承認 │  │ 却下 │
└─────┘  └──┬──┘
             │
        却下理由を記録
        member に通知
```

### 承認対象
- シフト希望の申請
- 休暇申請（有給・半休・欠勤）
- 打刻修正の申請

### ルール
- 承認権限は admin のみが持つ
- 却下時は理由の入力を必須とする
- 承認・却下の結果は申請者に即時反映される
- 承認済みの申請は取り消し不可（再申請で対応）

## 4. 役割（ロール）

### admin（管理者）
- チーム全員の勤怠データ閲覧
- 申請の承認・却下
- メンバーの日報閲覧
- ダッシュボードでチーム全体の稼働状況確認
- メンバーの招待・ロール変更

### member（一般メンバー）
- 自分の出退勤打刻
- シフト・休暇の申請
- 自分の日報作成・閲覧
- 自分のカレンダー・勤務サマリー確認

### 権限マトリクス

| 機能 | admin | member |
|------|-------|--------|
| 打刻（自分） | ○ | ○ |
| 打刻修正申請 | ○ | ○ |
| シフト・休暇申請 | ○ | ○ |
| 申請の承認・却下 | ○ | × |
| 自分のカレンダー | ○ | ○ |
| 他メンバーのカレンダー | ○ | × |
| 日報作成（自分） | ○ | ○ |
| 他メンバーの日報閲覧 | ○ | × |
| ダッシュボード（チーム） | ○ | × |
| ダッシュボード（自分） | ○ | ○ |
| メンバー招待・管理 | ○ | × |

## 5. 想定業務

### 現場
- 飲食店・小売店（シフト制の店舗スタッフ）
- 小規模IT企業（リモートワーク混在チーム）
- クリニック・美容院（少人数スタッフ）

### 利用者と使い方

**店舗スタッフ（member）の1日:**
1. 出勤時にスマホで打刻画面を開き「出勤」をタップ
2. 来週のシフト希望をシフト申請画面から提出
3. 退勤前に日報画面で業務内容を簡単に記入
4. 「退勤」をタップして帰宅

**店長・マネージャー（admin）の1日:**
1. 朝、ダッシュボードで本日の出勤状況を確認
2. 未処理の申請を承認 or 却下
3. カレンダーで今週のチームシフトを俯瞰
4. 月末にダッシュボードで勤務サマリーを確認・エクスポート

## 6. 技術スタック

- **フロントエンド**: Next.js (App Router) + TypeScript
- **UI**: Tailwind CSS + shadcn/ui
- **バックエンド**: Next.js API Routes (Route Handlers)
- **データベース**: Supabase (PostgreSQL)
- **認証**: Supabase Auth
- **デプロイ**: Render


---

## 7. テーブル設計

> **テーブル設計はAIに委任する。**
> 上記の画面構成・承認フロー・ロール定義・想定業務を満たすスキーマをAIが設計すること。
>
> 設計時の制約:
> - Supabase (PostgreSQL) を前提とする
> - Supabase Auth の `auth.users` と連携する
> - RLS (Row Level Security) を考慮した設計にする
> - タイムスタンプは UTC で保存し、表示時にJSTへ変換する
> - 論理削除（soft delete）を採用する

---

## 8. 開発ルール

- テーブル設計を最初に確定させてからUI実装に入る
- 各画面は独立したfeatureディレクトリで管理する
- API は `/api/v1/` 配下に REST で設計する
- 全てのユーザー入力は Zod でバリデーションする
- immutable なデータ操作を徹底する（スプレッド構文で新オブジェクト生成）
